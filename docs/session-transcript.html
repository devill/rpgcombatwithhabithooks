RPG Combat Kata Specification

Core Character Mechanics

Characters begin with 1000 health and are alive. They can inflict damage on other characters, with damage reducing health until reaching zero (death). Characters cannot damage themselves. Self-healing is permitted for living characters only.

Level System

Characters start at level 1. Health maximums remain at 1000 until level 6, when they increase to 1500.

Damage modifiers apply based on level differences: "If the target is 5 or more Levels above the attacker, Damage is reduced by 50%" and conversely, "If the target is 5 or more Levels below the attacker, Damage is increased by 50%"

Faction Mechanics

Characters may join multiple factions (starting unaffiliated). Those sharing a faction become allies—they cannot damage each other but can heal each other. Non-allies cannot provide healing.

Magical Objects

Magical Objects possess fixed maximum health. When depleted to zero, they are destroyed. Characters cannot heal these objects.

Healing Objects restore character health up to both the object's and character's maximums. Magical Weapons deal fixed damage per use while losing one health point per activation. Weapons cannot restore health.

Level Advancement

"Level 1 Characters that survive 1000 damage points gain a level" with damage accumulating across battles. Each subsequent level requires surviving additional damage equal to that level number times 1000.

Alternatively, characters gain levels by joining distinct factions: level 1 requires joining 3 factions; higher levels require joining 3 additional factions per level.

Maximum character level is 10 with no level loss possible.

Here's the phase plan. The kata's mechanics naturally stack, so phases are mostly sequential — each ends with a green pnpm run ci + commit per project rules.

Phase plan

Phase 1 — Core character: health, damage, death, healing

  • Character with health=1000, alive=true, attack(target, amount), heal(amount)
  • Death at 0 HP; healing capped at max HP; dead characters can't heal
  • Gate: tests for damage-reduces-HP, death-at-zero, heal-caps-at-max, dead-cannot-heal

Phase 2 — Self-targeting rules

  • Cannot damage self (silent no-op)
  • Self-healing only when alive (already implied — make explicit)
  • Gate: tests for attack(self) is no-op; heal() while dead is no-op

Phase 3 — Levels and damage modifiers

  • level starts at 1; max HP becomes 1500 at level ≥ 6
  • Target 5+ levels above attacker → damage × 0.5
  • Target 5+ levels below attacker → damage × 1.5
  • Gate: parametrized tests across level diffs (−6, −5, −4, 0, +4, +5, +6); max-HP check at level 6

Phase 4 — Factions

  • join(faction), leave(faction); characters start unaffiliated
  • Allies (shared faction) cannot damage each other; non-allies can heal only themselves
  • Gate: tests for ally-attack blocked, ally-heal allowed, stranger-heal blocked, self-heal still works

Phase 5 — Magical Objects

  • MagicalObject base: fixed max HP, takes damage, destroyed at 0, cannot be healed
  • HealingObject: restores a character's HP capped by both maxima
  • MagicalWeapon: fixed damage per use, loses 1 HP per activation; cannot heal
  • Gate: tests for object destruction, healing-cap on both sides, weapon HP decay to destruction, weapon-cannot-heal

Phase 6 — Level advancement

  • Track cumulative damage taken; thresholds: lvl 1→2 at 1000; lvl N→N+1 after N×1000 more (total 1000, 3000, 6000, …)
  • Faction-count path: lvl 1 unlocked at 3 distinct factions ever-joined; each additional level needs +3 distinct
  • Either path can trigger a level-up; cap at level 10; no level loss
  • Gate: tests for damage-threshold level-ups (incl. mid-fight), faction-path level-ups, lvl-10 cap, no level loss on faction-leave

Ambiguities I'd want to nail down before Phase 1

  1. "Cannot damage themselves" — silent no-op vs. throw? I'd default to silent.
  2. Faction-leveling joins — does the count track distinct factions ever joined (cumulative) or currently in? The spec is ambiguous. I'd default to cumulative since "no level loss possible."
  3. Healing others requires shared faction — the spec says "Non-allies cannot provide healing." Does that block healing yourself if you're unaffiliated? I read it as "no, self-heal is always allowed; the ally rule only gates healing others."

Parallelism notes

Phases are sequential (each builds on the previous domain model), but within each phase, test cases can be parallelised across sub-agents (e.g., Phase 3's level-diff matrix, Phase 5's three object types). Phase 6's two leveling paths are independent and can be implemented in parallel sub-tasks once the rest is in place.

Approved Fixture

Create or update an approved fixture test.

Concept

Approved fixtures are test files containing both input and expected output in a domain-specific format optimized for human review. The test harness parses these files, runs the code, and validates against the expected output.

Why: Validate test execution logic once, then adding tests is just reviewing fixture diffs.

Creating a New Fixture

  1. Identify the domain - What are you testing? (data transformation, visual algorithm, multi-step process, API sequence)
  1. Design the format - Create a human-readable structure for this domain:
  • Use markdown, ASCII art, or simple DSL
  • Input section clearly separated from expected output
  • Minimize noise, maximize signal
  1. Create the fixture file - Name it descriptively: {scenario}.approved.{ext}
  1. Write/update the test harness - If one doesn't exist for this format:
  • Parse the fixture file
  • Extract input and expected output
  • Run code with input
  • Compare actual vs expected
  • On failure, show diff or regenerate for review

Updating Existing Fixtures

  1. Find the fixture format used in the project (check test directories)
  2. Follow the existing structure exactly
  3. Add new scenarios as new fixture files or sections

Format Examples

Data transformation: markdown with input/output sections
Visual/spatial: ASCII art or generated images showing before/after
API sequences: structured list of calls and responses
State machines: transition tables

Key Principle

The fixture should be so clear that reviewing correctness is trivial - a quick visual scan or diff.

Format design — domain shape

The kata has a stable story shape: Setup → Actions → Expected. A character is fully described by hp, level, factions, and alive. Actions are short verbs over named characters/objects. That suggests a small narrative DSL — Gherkin-flavoured but tighter.

Proposed format

One scenario per file, named {scenario}.approved.md, with three fenced sections:

# scenario: heal cannot exceed max

## given
Alice: hp=950

## when
Alice heals 200

## then
Alice: hp=1000, alive

Defaults: omitted fields fall back to spec defaults (hp=1000, level=1, alive, factions=[]). So the trivial case is one line in given and one line in then.

Examples across the kata's complexity range

Phase 1 — overkill leaves health at 0:

## given
Alice: hp=100

## when
Bob attacks Alice for 500

## then
Alice: hp=0, dead

Phase 3 — level-diff damage modifier:

## given
Alice: level=10
Bob:   level=1

## when
Alice attacks Bob for 100

## then
Bob: hp=850          # 100 × 1.5

Phase 4 — allies can't damage each other:

## given
Alice: factions=[Red, Blue]
Bob:   factions=[Red]

## when
Alice attacks Bob for 500

## then
Bob: hp=1000, alive

Phase 5 — weapon decays per use:

## given
Sword: kind=weapon, hp=3, damage=100
Bob:   hp=1000

## when
Alice attacks Bob with Sword
Alice attacks Bob with Sword

## then
Bob:   hp=800
Sword: hp=1

Phase 6 — multi-step level-up:

## given
Alice: level=1

## when
Bob attacks Alice for 600
Bob attacks Alice for 500

## then
Alice: hp=0, dead, level=1   # died before crossing the threshold

(That one alone surfaces an interesting spec question: do you have to survive the damage, i.e. still be alive at threshold? Spec says "survive 1000 damage points" → yes. Fixture format makes this kind of question visible.)

Harness shape

A single Vitest file that:

  1. Globs tests/fixtures/**/*.approved.md.
  2. For each file, parses the three sections.
  3. given parser: one line per actor. Name: key=value, key=value, … → builds Character / MagicalObject instances and stashes them in a name→entity map.
  4. when parser: one action per line, matched against a small grammar:
  • A attacks B for N
  • A attacks B with Weapon
  • A heals N (self)
  • A heals B for N
  • A joins F / A leaves F
  1. then parser: same shape as given — asserts the listed fields match.
  2. On mismatch: write {scenario}.actual.md next to the approved file; print a unified diff. Reviewer compares the two files side-by-side and either fixes the code or mvs .actual over .approved.

One it() per fixture, named after the file. No magic — the assertion is just deep-equal on the projected state.

Where this pays off — and where it doesn't

Honest take: for Phase 1's 7 tests, hand-written unit tests are simpler. The fixture machinery (parser + harness) is ~80–120 lines of code that buys very little for trivial 2-line scenarios.

The format pays off from Phase 3 onward, where the parameter space explodes (level diffs × damages, faction combinations × heal/attack, three object types, multi-step level-up sequences). Adding a test becomes "write a 6-line markdown file" instead of "wire up a new TS test."

Suggested approach

  1. Phase 1: keep the 7 plain Vitest tests I listed. Get the core domain modelled with no parsing overhead.
  2. Between Phase 2 and Phase 3: introduce the fixture harness as a small focused change, port the existing tests as fixtures (cheap sanity check that the harness works), then use it from Phase 3 onward.
  3. As the grammar grows phase-by-phase, the harness grows with it — each new verb is a small addition to the when parser.

  level: number;

  alive = true;
